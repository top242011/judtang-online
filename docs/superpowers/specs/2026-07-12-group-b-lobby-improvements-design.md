**วันที่:** 2026-07-12

**ไม่รวมในสโคปนี้ (แยกไปทำทีหลัง):** กลุ่ม C (ความลึกของการเจรจา/บริหาร), กลุ่ม D (ศาลรัฐธรรมนูญ), กลุ่ม E (วิสัยทัศน์ระยะยาว)

## แนวคิดสถาปัตยกรรมหลัก

ตอนนี้ `PARTY_TEMPLATES` มี 6 พรรคตายตัว และ `handleCreateRoom()` สร้าง `slots` จากทั้ง 6 พรรคเสมอ ฟังก์ชันเกือบทั้งหมดของเกม (คำนวณเลือกตั้ง, coalition, cabinet, governing) วนลูป `r.slots` โดยตรง

แทนที่จะแก้ทุกจุดที่วนลูป `r.slots` ให้เช็ค "เปิดใช้งานไหม" ทุกครั้ง จะเพิ่ม field `enabled` ต่อ slot ที่มีผลใน **หน้า lobby เท่านั้น** แล้ว**กรองอาร์เรย์ `slots` ให้เหลือแต่พรรคที่ `enabled===true` ในจุดเดียว** คือตอน `startGame()` (lobby → campaign) หลังจากนั้นพรรคที่ถูกปิดจะหายไปจากอาร์เรย์ทั้งหมดตลอดทั้งรอบเกม ทุกฟังก์ชันดาวน์สตรีมทำงานเหมือนเดิมโดยไม่ต้องแก้ไข เพราะมันแค่วนลูปอาร์เรย์ที่สั้นลง

`MAJORITY`/`TOTAL_SEATS` มาจาก `REGIONS` (ที่นั่งตามภูมิภาค) ไม่ใช่จากจำนวนพรรค จึงไม่ได้รับผลกระทบจากจำนวนพรรคที่เปิดใช้งาน

## 1. เปิด/ปิดพรรค (ขั้นต่ำ 2 พรรค)

### Data model
เพิ่ม field ต่อ slot ใน `handleCreateRoom()` (บรรทัด 782-786):
```
enabled: boolean   // default true — พรรคนี้จะเล่นในรอบนี้ไหม
```

### UI (`slotCard()`, บรรทัด 507-518)
แยกเรนเดอร์ตามสถานะ:
- **ปิดใช้งาน (`enabled===false`)**: แถวจางลง (`opacity` ต่ำ), แสดง tag "ปิดใช้งาน" แทนปุ่มเข้าร่วม; ถ้าเป็นหัวห้องเห็นปุ่ม "เปิดพรรคนี้"
- **เปิดใช้งานตามปกติ**: เหมือนเดิม แต่ถ้าเป็นหัวห้องและ slot นี้ยังไม่มีคนจับจอง (`!s.claimedBy`) จะมีปุ่มเพิ่ม "ปิดพรรคนี้" ต่อท้าย

### Logic
เพิ่มฟังก์ชัน `toggleSlotEnabled(idx)`:
- ทำงานเฉพาะ `isHost()`
- ปิดได้เฉพาะ slot ที่ `!s.claimedBy` (slot ที่มีคนจับจองต้อง "เตะออก" ก่อน — ดูข้อ 2)
- ก่อนปิด เช็คว่าจำนวน slot ที่ `enabled===true` หลังปิดจะไม่ต่ำกว่า 2 — ถ้าต่ำกว่าให้ปุ่ม "ปิดพรรคนี้" เป็น `disabled` ตั้งแต่ตอน render (ไม่ต้องรอ error ตอนกด)
- เปิดกลับได้เสมอไม่มีเงื่อนไข

### `startGame()` (บรรทัด 831-834)
เปลี่ยนจาก reset ทุก slot เป็น**กรองก่อน reset**:
```javascript
function startGame(){
  const slots=ui.room.slots.filter(s=>s.enabled).map(s=>({...s,submitted:false,budget:{north:0,isaan:0,central:0,bangkok:0,south:0}}));
  roomRef().update({phase:'campaign',slots});
}
```
พรรคที่ `enabled===false` จะหายไปจากอาร์เรย์ `slots` ที่ Firestore ตั้งแต่จุดนี้ — ไม่ต้องมีการเช็ค `enabled` อีกในฟังก์ชันอื่นใดเลยตลอดทั้งเกม

**Edge case:** ต้องคง gate เดิม `humanCount<2` (จำนวนคนเล่นจริง) ไว้คู่กับ gate ใหม่เรื่องจำนวนพรรคเปิดใช้งานขั้นต่ำ 2 พรรค — ทั้งสองเงื่อนไขเป็นคนละเรื่องกัน (จำนวนคนจริง vs จำนวนพรรคในกระดาน) และต้องผ่านทั้งคู่ปุ่ม "เริ่มเกม" ถึงจะกดได้

## 2. ล็อคสล็อตเป็น AI / เตะผู้เล่นออกกลับเป็น AI

### Data model
เพิ่ม field ต่อ slot:
```
aiLocked: boolean   // default false — ถ้า true พรรคนี้ AI เล่นเสมอ ห้ามคนจับจอง
```

### UI
- Slot ที่ `enabled===true && !s.claimedBy`: ถ้าเป็นหัวห้องเห็นปุ่มเพิ่ม "ล็อคเป็น AI" (ถ้า `!aiLocked`) หรือ "ปลดล็อค AI" (ถ้า `aiLocked`)
- Slot ที่ `aiLocked===true`: ซ่อนปุ่ม "เข้าร่วมพรรคนี้" สำหรับทุกคน (รวมหัวห้อง) แสดง tag "AI เท่านั้น" แทน
- Slot ที่มีคนจับจองอยู่ (`s.claimedBy` และ `s.claimedBy!==MY_ID`): ถ้าเป็นหัวห้องเห็นปุ่ม "เตะออก" — **หัวห้องเตะตัวเองไม่ได้** (ปุ่มนี้ไม่แสดงถ้า `s.claimedBy===MY_ID`)

### Logic
เพิ่มฟังก์ชัน `toggleSlotAiLock(idx)`:
- ทำงานเฉพาะ `isHost()` และเฉพาะ slot ที่ `!s.claimedBy`
- สลับค่า `aiLocked`

เพิ่มฟังก์ชัน `kickSlot(idx)`:
- ทำงานเฉพาะ `isHost()` และเฉพาะ slot ที่มีคนจับจองอยู่และไม่ใช่ตัวเอง
- ตั้ง `claimedBy:null, playerDisplayName:null, ready:false, aiLocked:true` (ตามที่ตกลงไว้ว่า "เตะแล้วกลับเป็น AI" ไม่ใช่เปิดให้คนอื่นจับจองต่อทันที — หัวห้องต้องกดปลดล็อค AI เองถ้าอยากเปิดให้คนอื่นจับจองใหม่)

**Guard เพิ่มใน `claimSlot(idx)`** (บรรทัด 820-825): เพิ่มการเช็ค `if(!slots[idx].enabled || slots[idx].aiLocked) return;` ที่ต้นฟังก์ชัน เป็นการป้องกันสองชั้น (defense in depth) เผื่อ UI ไม่ทันซิงค์ ถึงแม้ปุ่มจะถูกซ่อนไปแล้วในกรณีปกติ

## 3. ปุ่ม Ready (บังคับทุกคน)

### Data model
เพิ่ม field ต่อ slot:
```
ready: boolean   // default false
```

### UI (ในส่วน "ตั้งชื่อพรรคของคุณ" ที่แสดงเฉพาะ `mine`, บรรทัด 489-499)
เพิ่มปุ่ม toggle "พร้อมแล้ว" / "ยังไม่พร้อม" ต่อท้ายส่วนตั้งชื่อ/เลือกสี

ใน `slotCard()`: สำหรับ slot ที่มีคนจับจองอยู่ (ทุกคนเห็น ไม่ใช่แค่หัวห้อง) แสดง tag เพิ่มข้าง tag เดิม — "พร้อมแล้ว" (เขียว) หรือ "ยังไม่พร้อม" (เหลือง) ตาม `s.ready` เพื่อให้ทุกคนเห็นว่าใครทำให้เริ่มเกมไม่ได้

`doc-sub` ของหน้า lobby (บรรทัด 486) เปลี่ยนจาก `ผู้เล่น ${humanCount} / ${r.slots.length} คน` เป็นแสดงทั้งจำนวนคนเล่นและจำนวนที่พร้อมแล้ว เช่น `ผู้เล่น ${humanCount} คน · พร้อมแล้ว ${readyCount}/${humanCount}`

### Logic
เพิ่มฟังก์ชัน `toggleMyReady()`:
```javascript
function toggleMyReady(){
  const slots=ui.room.slots.map(s=>s.claimedBy===MY_ID?{...s,ready:!s.ready}:s);
  roomRef().update({slots});
}
```

**แก้ `updateMySlotField()`** (บรรทัด 827-830): ทุกครั้งที่แก้ชื่อหรือสี (สอง use-site เดียวที่เรียกฟังก์ชันนี้ในโค้ดทั้งไฟล์) ต้องรีเซ็ต `ready:false` โดยอัตโนมัติ:
```javascript
function updateMySlotField(field,val){
  const slots=ui.room.slots.map(s=>s.claimedBy===MY_ID?{...s,[field]:val,ready:false}:s);
  roomRef().update({slots});
}
```

**แก้ปุ่ม "เริ่มเกม"** ใน `PHASE_TEMPLATES.lobby` (บรรทัด 502): เพิ่มเงื่อนไข gate ใหม่ — ทุก slot ที่ `s.claimedBy` (claimed จริง ไม่ใช่ aiLocked) ต้องมี `s.ready===true` ครบทุกคน ถึงจะกดปุ่มได้ (ควบคู่กับ gate เดิม `humanCount<2` และ gate ใหม่จากข้อ 1 เรื่องจำนวนพรรคเปิดใช้งานขั้นต่ำ 2 พรรค — ต้องผ่านทั้ง 3 เงื่อนไข)

## Field defaults ที่ต้องอัปเดตให้ครบทุกจุดสร้าง/รีเซ็ตห้อง

`enabled`, `aiLocked`, `ready` เป็น field ใหม่ ต้องเช็คทุกจุดที่สร้างหรือรีเซ็ต slots:

- **`handleCreateRoom()`** (บรรทัด 782-786): ต้องใส่ `enabled:true, aiLocked:false, ready:false` ในทุก slot ตอนสร้างห้องใหม่
- **`resetRoomToLobby()`** (บรรทัด 1195-1200, มาจาก Group A): สร้าง slot ใหม่ด้วย `{...s, policy:..., submitted:false, budget:...}` — เพราะใช้ `...s` spread จึงคง `enabled`/`aiLocked` ของหัวห้องที่ตั้งไว้ก่อนหน้าไว้อัตโนมัติ (ไม่รีเซ็ตกลับเป็นค่าเริ่มต้น เป็นพฤติกรรมที่ตั้งใจ — หัวห้องไม่ต้องตั้งค่าใหม่ทุกครั้งที่กด reset) **แต่ต้องเพิ่ม `ready:false` เข้าไปในอ็อบเจกต์เดียวกัน** เพื่อบังคับให้ทุกคนกด Ready ใหม่หลัง reset
- **`claimSlot()`** (บรรทัด 820-825): slot ที่เพิ่งถูกจับจองต้องเริ่มด้วย `ready:false` เสมอ (ค่าเริ่มต้นอยู่แล้วเพราะ `kickSlot`/ค่าเริ่มต้นตอนสร้างห้องตั้งไว้เป็น false — ไม่ต้องเซ็ตซ้ำเพราะ `claimSlot` ไม่ได้แตะ field `ready` เลย มันคงค่าที่มีอยู่เดิมของ slot นั้นไว้)

## Edge cases

- หัวห้องปิด/ล็อค AI/เตะคนออกได้เฉพาะตอนอยู่ใน `phase:'lobby'` เท่านั้น (ปุ่มเหล่านี้อยู่ใน `PHASE_TEMPLATES.lobby`/`slotCard()` ซึ่งเรนเดอร์เฉพาะตอน lobby อยู่แล้วโดยธรรมชาติของโครงสร้างโค้ด ไม่ต้อง guard เพิ่ม)
- ถ้าหัวห้องเตะผู้เล่นที่กำลังจะกดปุ่ม claim/ready พร้อมกันพอดี (race condition) — ไม่อยู่ในสโคปนี้ ยอมรับความเสี่ยงเดียวกับที่มีอยู่แล้วในฟังก์ชัน `claimSlot`/`updateMySlotField` เดิม (ไม่ใช้ Firestore transaction เหมือนที่ยอมรับใน Group A)
- จำนวนพรรคเปิดใช้งานขั้นต่ำ 2 พรรค บังคับที่ปุ่ม "ปิดพรรคนี้" (disabled ล่วงหน้า) ไม่ใช่ error ตอนกด — ป้องกันไม่ให้ห้องเข้าสู่สถานะที่เล่นไม่ได้ตั้งแต่ต้น

## Testing Plan (manual, ไม่มี automated test harness ในโปรเจกต์นี้)

1. หัวห้องปิดพรรคที่ยังไม่มีคนจับจอง 4 จาก 6 พรรค (เหลือ 2) → ปุ่ม "ปิดพรรคนี้" ของอีก 2 พรรคที่เหลือต้อง disabled ทันที; กด "เริ่มเกม" แล้วเช็คว่าหน้า campaign/election เหลือแค่ 2 พรรคที่เปิดใช้งาน
2. หัวห้องล็อค AI พรรคที่ว่างอยู่ 1 พรรค → ปุ่ม "เข้าร่วมพรรคนี้" ของพรรคนั้นหายไปสำหรับทุกคน (ทดสอบจาก client อื่นที่ไม่ใช่หัวห้องด้วย); ปลดล็อคแล้วปุ่มกลับมา
3. ผู้เล่น B จับจองพรรคหนึ่ง แล้วหัวห้องกด "เตะออก" → ฝั่ง B เห็นตัวเองหลุดจาก slot กลับไปเป็น "ยังไม่ได้เลือกพรรค"; slot นั้นกลายเป็น AI-locked ทันที (ปุ่มเข้าร่วมหายไป) จนกว่าหัวห้องจะปลดล็อค
4. ผู้เล่น 2 คนจับจองพรรค ตั้งชื่อ/สี แต่ยังไม่กด Ready → ปุ่ม "เริ่มเกม" ของหัวห้อง disabled; กด Ready ครบทั้งคู่ → ปุ่มเปิดใช้งาน; แก้ชื่อพรรคหลังกด Ready แล้ว → สถานะ Ready รีเซ็ตกลับเป็นยังไม่พร้อม และปุ่มเริ่มเกม disabled อีกครั้ง
5. กด "รีเซ็ตเกม" (จาก Group A) กลางเกม แล้วเช็คว่า `enabled`/`aiLocked` ของแต่ละ slot ยังคงค่าที่ตั้งไว้ก่อน reset แต่ `ready` ทุก slot กลับเป็น false
